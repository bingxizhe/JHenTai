## 项目概览

JHenTai 是一个跨平台（Android/iOS/Windows/macOS/Linux）的 E-Hentai/EXHentai 漫画阅读器，版本 `8.0.13+315`，基于 Flutter + GetX 构建。

## 下载系统架构总览

项目有 **两套独立的下载引擎**，分别处理不同类型的下载：

| 服务 | 文件 | 行数 | 用途 |
|------|------|------|------|
| `GalleryDownloadService` | `gallery_download_service.dart` | 1847 | **逐张图片下载**（解析缩略图页→图片页→图片URL→下载） |
| `ArchiveDownloadService` | `archive_download_service.dart` | ~1300 | **归档整包下载**（解锁→获取下载URL→多Isolate下载ZIP→解压） |

## 一、Gallery 下载引擎（核心）

### 1.1 数据结构

```
GalleryDownloadInfo (内存中的运行时状态)
├── thumbnailsCountPerPage  // 每页缩略图数量(20/40)
├── tasks: List<AsyncTask>  // 提交到 EHExecutor 的任务引用
├── cancelToken: CancelToken // Dio 取消令牌
├── downloadProgress        // 下载进度
│   ├── curCount / totalCount
│   ├── downloadStatus (none/switching/paused/downloading/downloaded/failed)
│   └── hasDownloaded: List<bool>  // 每张图片的下载状态
├── imageHrefs: List<GalleryThumbnail?>  // 缩略图页URL（第3步解析结果）
├── images: List<GalleryImage?>          // 图片信息（第4步解析结果）
├── speedComputer         // 速度计算器
├── priority / sortOrder / group
```

### 1.2 下载流程（3阶段流水线）

每张图片的下载是一个 **3阶段流水线**，每个阶段都是独立的异步任务提交到 `EHExecutor`：

```
阶段1: _parseImageHrefTask    解析缩略图页，获取图片页URL (href)
   ↓
阶段2: _parseImageUrlTask     解析图片页，获取图片直接下载URL
   ↓
阶段3: _downloadImageTask     下载图片文件到磁盘
```

**关键逻辑**（`_processImage` 方法，第975行）：
- 如果图片已下载 → 跳过
- 如果已有 URL（阶段2已完成）→ 直接提交下载任务
- 如果已有 href（阶段1已完成）→ 提交 URL 解析任务
- 如果都没有 → 提交 href 解析任务

### 1.3 EHExecutor（优先级任务调度器）

自定义的优先级任务调度器（`eh_executor.dart`），基于 `executor` 包改造：

- **优先级队列**：替换原始 ListQueue 为 `PriorityQueue`，支持按优先级调度
- **并发控制**：`concurrency` 控制最大同时运行任务数（默认6）
- **速率限制**：`Rate(maximum, period)` 控制任务启动频率（默认2个/秒）
- **任务取消**：`cancelTask()` 从等待队列中移除任务

**优先级计算**（第800-820行）：
```
画廊优先级 = userPriority × 1亿 + (insertTime编码 × 2000) + serialNo
```
- 高优先级画廊先下载
- 同优先级按插入时间排序（`downloadAllGallerysOfSamePriority=false` 时逐个下载）
- 同画廊内按 serialNo 顺序下载

### 1.4 下载状态机

```
DownloadStatus:
  none → downloading → downloaded
                  ↘ paused → downloading (恢复)
                  ↘ downloadFailed
```

**暂停**（第167行）：取消 CancelToken → 清空 tasks → 所有 downloading 的图片标记为 paused → 保存 metadata
**恢复**（第216行）：新建 CancelToken → paused 图片标记为 downloading → 重新调用 `downloadGallery(resume: true)`
**删除**（第259行）：暂停 → 删除超分辨率记录 → 清数据库 → 可选删除磁盘文件 → 清内存

### 1.5 三层持久化

| 层级 | 机制 | 内容 |
|------|------|------|
| **数据库** | Drift (SQLite) | `gallery_downloaded_v2` 表 + `image` 表 |
| **磁盘文件** | `{downloadDir}/{gid} - {title}/metadata` | JSON 格式，含画廊信息+所有图片状态 |
| **ComicInfo.xml** | 同上目录 | 漫画阅读器通用元数据格式 |

### 1.6 错误处理与重试

- 每个阶段最多重试 3 次（`_maxRetryTimes`）
- 下载失败后自动 **重新解析URL**（`_reParseImageUrlAndDownload`），因为 URL 可能已过期
- 下载内容检测（`imageData2Exception`）：空白图片、高峰时段GP不足、无效token、H@H节点错误等
- 403 响应（H@H 死节点）→ 重新解析
- 410/404 → 暂停并提示

### 1.7 画廊更新机制

`updateGallery`（第274行）：获取新版本详情 → 创建新下载任务 → 保留 `oldVersionGalleryUrl`
- 下载时通过 `imageHash` 匹配新旧图片，**相同的图片直接复制**，避免重复下载
- 支持从 JHenTai 服务器获取图片哈希（`useJH2UpdateGallery`）加速匹配

### 1.8 缓存优先

`_tryLoadFromCacheInsteadDownload`（第1410行）：下载前先检查 `extended_image` 的缓存，命中则直接复制，跳过网络请求。

## 二、Archive 下载引擎

### 2.1 5阶段流程

```
阶段1: _unlock               请求解锁归档
   ↓
阶段2: _getDownloadPageUrl    轮询等待解锁完成，获取下载页URL
   ↓
阶段3: _getDownloadUrl        解析下载页，获取ZIP下载URL
   ↓
阶段4: _doDownloadArchiveViaMultiIsolate  多Isolate下载ZIP文件
   ↓
阶段5: _unpackingArchive      解压ZIP到目录
```

### 2.2 Archive 状态机

```
ArchiveStatus:
  unlocking → unlocked → parsingDownloadPageUrl → parsedDownloadPageUrl 
    → parsingDownloadUrl → parsedDownloadUrl → downloading → downloaded 
    → unpacking → completed
  任意阶段可 → paused / waitingIsolate / needReUnlock
```

### 2.3 两种解析源

| 源 | 说明 |
|---|---|
| `ArchiveParseSource.official` | 官方EH归档，需解锁→轮询→解析下载URL |
| `ArchiveParseSource.bot` | ArchiveBot 第三方服务，直接解析下载URL |

### 2.4 多Isolate下载

使用自定义 `JDownloader`（`jiangtian616/JDownloader` fork）：
- 支持多 Isolate 并行下载同一个文件（分片）
- 全局 Isolate 上限 10 个（`_maxIsolateCountsTotal`）
- 超出上限的任务进入 `waitingIsolate` 状态，等待唤醒
- 410/404 时自动检测原因（下载量过大/IP配额耗尽/会话过期），决定是否需要重新解锁

### 2.5 归档转画廊

`migrate2Gallery`（第260行）：已完成的归档可转换为画廊下载记录，复用画廊阅读器的功能。

## 任务一：启动时本地画廊文件扫描流程分析

### 启动调用链

```
main() 
  → 遍历 lifeCircleBeans 按拓扑序调用 initBean()
    → localGalleryService.doInitBean()   (main.dart:64, 第117行 await)
      → await refreshLocalGallerys()     (local_gallery_service.dart:39)
        → await _loadGalleriesFromDisk() (local_gallery_service.dart:59)
          → 遍历 extraGalleryScanPath 中的每个路径
            → 递归调用 _parseDirectory() 扫描目录树
```

### 关键性能问题

1. **同步阻塞主线程**：`initBean()` 在 `main()` 中被 `await` 串行调用（`main.dart:118`），`localGalleryService.doInitBean()` 内部 `await refreshLocalGallerys()`，整个扫描过程在 `runApp()` 之前完成，**直接阻塞 UI 渲染**。

2. **大量同步文件 I/O**：`_parseDirectory()` 中虽然使用了 `directory.list()` (异步)，但内部逻辑密集：
   - 每个目录检查 `metadata` 文件是否存在（2次 `File.exists()`）
   - 遍历目录下所有文件判断是否为图片（`FileUtil.isImageExtension`）
   - 递归扫描所有子目录
   - 使用 `Completer` 手动管理异步，但实际回调链在主线程 Isolate 执行

3. **`restoreTasks()` 同样阻塞**：`galleryDownloadService.doInitBean()` 和 `archiveDownloadService.doInitBean()` 中，若 `restoreTasksAutomatically=true`，会同步扫描下载目录读取 metadata 文件，使用 `listSync()`（同步！）遍历目录。

4. **无进度反馈**：扫描期间 UI 尚未渲染，用户看到的是白屏，无任何进度提示。

### 影响量化

- 扫描时间与目录数量、文件数量正相关
- `extraGalleryScanPath` 可能有多个路径
- 每个画廊目录至少 2 次 `File.exists()` + 1 次 `directory.list()`
- 数百个画廊时，启动延迟可达数秒

## 任务二：延迟扫描方案可行性评估

### 方案描述

将 `LocalGalleryService` 的启动扫描从 `doInitBean()` 中移除，改为：
- 启动时仅初始化服务注册，不扫描文件
- 用户切换到"本地画廊"Tab 时，触发异步后台扫描
- UI 层实时显示扫描状态（loading/进度），扫描完成后自动刷新

### 可行性分析

| 维度 | 评估 |
|------|------|
| **功能正确性** | ✅ 完全可行。`LocalGalleryService` 的数据仅被本地画廊页面消费，启动时无其他服务依赖其扫描结果 |
| **启动性能** | ✅ 显著改善。移除 `await refreshLocalGallerys()` 后，`localGalleryService.doInitBean()` 几乎瞬时完成 |
| **用户体验** | ✅ 改善。启动白屏时间缩短；进入本地画廊页时有明确的加载状态反馈 |
| **代码侵入性** | ✅ 低。主要修改集中在 `LocalGalleryService` 和本地画廊页面，不影响其他服务 |
| **边界情况** | ⚠️ 需处理：用户快速切换 Tab 时防止重复扫描；扫描中退出页面的取消逻辑 |

### 与原方案对比

| | 原方案（启动扫描） | 新方案（延迟扫描） |
|---|---|---|
| 启动耗时 | 长（阻塞） | 短（跳过） |
| 首次进入本地页 | 即时显示 | 需等待扫描（有 loading 反馈） |
| 内存占用 | 启动即占用 | 按需占用 |
| 用户感知 | 白屏等待 | 快速启动 + 页面内加载 |
| 适用场景 | 画廊少时无感 | 画廊多时优势明显 |

**结论：方案可行，应实施。**

## 变更总结

### 修改了 3 个文件，共 25 行新增、2 行删除：

#### 1. `lib/src/service/local_gallery_service.dart`
- **移除** `doInitBean()` 中的 `await refreshLocalGallerys()`，启动时仅注册服务，不再扫描文件
- **新增** `_hasScanned` 标志位，记录是否已完成过扫描
- **新增** `ensureScanned()` 方法：若未扫描过则触发扫描，已扫描则立即返回
- **修改** `refreshLocalGallerys()`：扫描开始时重置 `_hasScanned=false`，完成时设置 `_hasScanned=true`

#### 2. `lib/src/pages/download/list/local/local_gallery_list_page_logic.dart`
- **新增** `onInit()` 生命周期方法，调用 `localGalleryService.ensureScanned()`
- 用户首次进入本地画廊列表页时触发异步扫描

#### 3. `lib/src/pages/download/grid/local/local_gallery_grid_page_logic.dart`
- **新增** `onInit()` 生命周期方法，调用 `localGalleryService.ensureScanned()`
- 用户首次进入本地画廊网格页时触发异步扫描

### 技术保障

| 保障项 | 实现方式 |
|--------|----------|
| **防止重复扫描** | `ensureScanned()` 检查 `_hasScanned`；`refreshLocalGallerys()` 检查 `loadingState == LoadingState.loading` |
| **UI 不卡死** | 扫描在 `onInit()` 中异步触发（不 await），UI 先渲染 `LoadingStateIndicator`（idle 状态默认显示加载动画），扫描完成后通过 `update([galleryCountChangedId])` 自动刷新 |
| **状态实时更新** | 复用现有 `GetBuilder<LocalGalleryService>` + `galleryCountChangedId` 机制，扫描完成自动触发 UI 重建 |
| **手动刷新兼容** | `handleRefreshLocalGallery()` 和设置页更改路径后的 `refreshLocalGallerys()` 调用均不受影响 |
